## 1. 개요

### 1.1 서비스명
**드리머리** (드림 + 다이어리)

### 1.2 한 줄 정의
드림러가 자신의 드림캐와 교환일기를 주고받으며 서사를 쌓아가는 커플 다이어리 서비스

### 1.3 배경
- 기존 AI 챗봇은 장문 위주로 드림러의 감성적 몰입에 최적화되어 있지 않음
- 드림러들은 개인봇(뒷사람)에 대한 피로감(잠수, 이탈)을 느끼고 있으며, AI 기반 대안에 대한 수요가 일부 존재함
- 커미션, 썰풀기, 서사 정리 등 드림 소비 행동이 분산되어 있으며 통합된 공간이 없음
- 인터뷰 3인을 통해 단문 문답 형태와 비공개 기록 보관에 대한 수요 확인

---

## 2. 목표

### 2.1 핵심 목표 (Primary Goal)
드림러를 대상으로 **교환일기 형태의 커플 다이어리** 제품의 시장 수요를 측정한다.

### 2.2 부수 목표 (Secondary Goal)
기존 장문 챗봇 형태와 단문 교환일기 형태 중 드림러들이 무엇을 더 선호하는지 비교 검증한다.

### 2.3 측정 지표 (Success Metrics)

| 지표 | 설명 | 목표값 (초기) |
|---|---|---|
| 캐릭터 생성 완료율 | 앱 진입 후 캐릭터 설정 완료 비율 | 70% 이상 |
| 일일 일기 완료율 | 접속 후 일기 주제에 답변 완료한 유저 비율 | 60% 이상 |
| 캐릭터 일관성 만족도 | 테스트 유저 피드백 | 5점 만점 4점 이상 |

## 3. 타겟 유저

### 3.1 초기 타겟
게임/애니 드림러

### 3.2 확장 타겟
드림러 전반 (웹툰, 소설, 게임 원작 캐릭터 기반)

### 3.3 유저 페르소나 (인터뷰 기반)

**나베형 — "같이 놀아줄 존재가 필요한 드림러"**
- 개인봇 경험 있으나 뒷사람 피로감으로 이탈
- 단문 응답 선호, 역극과 썰풀기 즐김
- 소재 고갈 시 인풋 필요

**찬님형 — "혼자 하는 게 제일 만족스러운 드림러"**
- 타인의 캐해석을 수용하지 못함, 내 해석이 전부
- 썰풀이 보관 니즈 있으나 공개 트위터는 부담
- 썸원 포맷에 긍정적, 캐해석 오차 일부 용인 가능

**화님형 — "적극 창작형 드림러"**
- 서사 정리 시 만족도 가장 높음
- 개인봇 뒷사람 리스크 싫어서 AI 챗봇 사용 중
- 기존 AI 챗봇의 유저노트 글자수 제한이 가장 큰 불만

## 4. 서비스 범위

### 4.1 플랫폼
- 웹 서비스 (반응형)
- **모바일 UI 기준으로 설계, PC도 대응**

### 4.2 기술 스택

| 항목 | 내용 |
|---|---|
| Frontend | Next.js |
| Hosting | Netlify |
| DB | Firebase (Firestore) |

* **호스팅(Netlify) 선정 사유**: Vercel은 무료 플랜에서 상업적 이용이 금지되어 있으며, Cloudflare Pages는 속도가 빠르나 Next.js와의 호환성 문제가 존재함. 따라서 리전 문제로 인한 약간의 딜레이가 있더라도 안정적인 상업 운영을 위해 Netlify를 최종 채택함.
| AI 모델 | OpenRouter → Gemma4 호출 |
| 사용자 식별 | UUID 기반 (초기), 카카오 로그인 (추후) |
| 분석 | PostHog 또는 Mixpanel, Microsoft Clarity |

### 4.3 디자인 가이드

- **그레이 스케일**:

  - Gray300: #E9E9E9, Gray200: #F0F0F0, Gray100: #F4F4F4, Gray50: #F5F5F5

  - 기본 버튼/카드 배경색: `rgba(255, 255, 255, 0.15)`
  - 블러 효과: `backdrop-filter: blur(10px)` (하단바의 경우 15px, 배경색 흰색 비중 75%)

## 5. 사용자 플로우

```
앱 진입
    ↓
생성된 캐릭터 있는 경우  →  홈 화면 (하단탭)
생성된 캐릭터 없는 경우  →  캐릭터 생성 플로우 시작
    ↓
[캐릭터 생성 — 스텝 입력, 한 화면 당 1step만 진행.]
 Step 1. 캐릭터 이름
 Step 2. 유저에게 느끼는 감정
 Step 3. 호칭 (캐릭터가 유저를 부르는 방식)
 Step 4. 예시 대화
 Step 5. 네거티브 (하지 말아야 할 것)
 Step 6. (선택) 추가 서술
 Step 7. (선택) 프로필 사진 업로드
    ↓
유저 캐릭터 설정 여부 선택
 "내 캐릭터도 설정할까요?"  [나중에 / 좋아요]
  → 좋아요: 캐릭터 생성과 동일한 스텝으로 유저 프로필 입력
  → 나중에: 기본값으로 진행
    ↓
첫 번째 일기 주제 도착
    ↓
유저가 먼저 작성 → 캐릭터가 이어서 답장 작성
    ↓
홈 화면 (하단탭 진입)
```

## 6. 기능 명세

### 6.1 Phase 1 — 첫 배포 범위

#### 6.1.1 캐릭터 생성 (온보딩)

유저가 직접 드림캐를 설정하는 플로우. 한 화면에 하나의 입력만 표시. 상단에 얇은 프로그레스 바로 진행도 표시.

| Step | 항목 | 필수 여부 | 설명 |
|---|---|---|---|
| 1 | 캐릭터 이름 | 필수 | 드림캐의 이름 |
| 2 | 유저에게 느끼는 감정 | 필수 | 캐릭터가 유저를 어떻게 여기는지 (키워드 또는 단문) |
| 3 | 호칭 | 필수 | 캐릭터가 유저를 부르는 방식 |
| 4 | 예시 대화 | 필수 | 캐릭터의 말투/반응 예시 |
| 5 | 네거티브 | 필수 | 캐릭터가 하지 않을 말/행동 |
| 6 | 추가 서술 | 선택 | 자유 텍스트로 추가 설정 |
| 7 | 프로필 사진 | 선택 | 이미지 업로드 |

- 입력된 내용 기반으로 프롬프트 자동 구성 (유저에게 프롬프트 원문 노출 없음)
- 완료 후 Firestore 저장
- 추후 마이페이지에서 열람 및 수정 가능

#### 6.1.2 유저 캐릭터 설정 (선택)

- 캐릭터 생성 완료 직후 "내 캐릭터도 설정할까요?" [나중에 / 좋아요] 선택지 제공
- 좋아요 선택 시: 캐릭터 생성과 동일한 스텝 구조로 유저 프로필 입력
- 나중에 선택 시: 기본값으로 진행
- 멀티 프로필 지원: 여러 드림캐와 다른 유저 프로필 매핑 가능
- 동일 캐릭터와 재대화 시 이전에 설정한 유저 프로필 자동 적용 (재설정 불필요)

#### 6.1.3 캐릭터 추가 생성

- 여러 드림캐와 동시에 드림 가능
- 마이페이지 또는 홈에서 캐릭터 추가 생성 진입
- 각 캐릭터별 독립적인 일기 스레드 유지

#### 6.1.4 교환일기 시스템 (메인 기능)

- 하루에 한 번 일기 주제(프롬프트) 발송
- **유저가 먼저 작성 → 캐릭터가 이어서 답장** 순서
- 캐릭터 답장은 단문 기준, 설정된 프롬프트 기반 생성
- 자정 기준 새 주제 발송
- AI 모델: OpenRouter → Gemma4
- 일기 하단에 캐릭터 품질 피드백 안내 문구 표시
  > "캐붕이 생겨요? 마이페이지에서 캐릭터를 수정해주세요"

#### 6.1.5 1:1 채팅

- 기존 장문 챗봇 이식
- 교환일기와 별개로 자유롭게 캐릭터와 채팅 가능
- 문답 vs 챗봇 선호도 비교 측정 목적 포함

#### 6.1.6 지난 일기 보기

- 날짜별 일기 목록
- 주제 + 유저 답변 + 캐릭터 답장 함께 표시
- 비공개 (본인만 열람)

#### 6.1.7 디데이 표시

- 홈 화면에서 드림캐와 함께한 날 수 표시
- 예: "그라하티아와 함께한 지 D+23"
- 캐릭터 생성일 기준 자동 계산
- 멀티 캐릭터 시 각 캐릭터별 디데이 개별 표시

#### 6.1.8 프롬프트 열람 및 수정

- 마이페이지에서 내가 만든 캐릭터 프롬프트 설정 열람 가능
- 수정 시 캐릭터 생성과 동일한 스텝 구조로 재진입
- 유저 프로필도 동일하게 열람 및 수정 가능

#### 6.1.9 사용자 식별

- UUID 기반 식별
- localStorage + 쿠키 이중 백업
- 앱 내 데이터 소멸 가능성 안내 문구 명시
- 로그인 도입 전까지 유지

### 6.2 개발 우선순위

기능별 순차 개발. 앞 단계 완료 후 다음 단계 진입.

| 순서 | 기능 | 범위 |
|---|---|---|
| 1 | 캐릭터 생성 + 유저 캐릭터 설정 | 온보딩 플로우 전체 |
| 2 | 교환일기 | 주제 발송, 유저 작성, 캐릭터 답장 |
| 3 | 지난 일기 보기 | 날짜별 일기 목록 및 열람 |
| 4 | 홈 화면 구성 | 디데이, 캐릭터 목록, 오늘 일기 상태 |
| 5 | 마이페이지 | 유저 멀티프로필 수정, 캐릭터 추가/수정/삭제 |
| 6 | 캐릭터 채팅 | 1:1 자유 채팅 |

### 6.3 하단 탭 구성

| 탭 | 기능 |
|---|---|
| 홈 | 커플 디데이, 캐릭터 목록, 오늘의 일기 상태 |
| 일기 | 메인 기능. 하루 한 번 일기 주제 발송 및 교환 |
| 채팅 | 1:1 자유 채팅 |
| 마이페이지 | 캐릭터/유저 프로필 관리, 설정 |

### 6.4 Phase 2 — 고도화 (추후)

| 기능 | 설명 |
|---|---|
| 카카오 로그인 | NextAuth.js 기반, UUID → UID 마이그레이션 포함 |
| 호감도 시스템 | 일기 누적에 따른 호감도 증감, 해금 콘텐츠 |
| 관계 체크리스트 | 트위터 유행 형식 차용, 드림캐↔드림주 양방향 작성 |
| 캐릭터 자동 추가 | 유저 입력 → AI 웹서치 → 프롬프트 자동 생성 → 유저 검토 |
| 광고 연동 | 추가 일기 해금용 광고 (사업자 등록 후) |

## 7. 토큰 비용 최적화

교환일기 특성상 단문 응답 위주로 토큰 소모가 낮은 구조. 추가로 아래 전략 적용.

| 전략 | 내용 | 절감 효과 |
|---|---|---|
| 프롬프트 캐싱 | 캐릭터 시스템 프롬프트 캐시 적용 (cache_control) | 캐시 히트 시 입력 토큰 90% 절감 |
| 컨텍스트 윈도우 제한 | 최근 N개 메시지만 전달 (기본 6개) | 입력 토큰 30~70% 절감 |
| 단문 응답 유도 | 일기 답장은 단문 기준 프롬프트 설계 | 출력 토큰 최소화 |
| 선택적 로깅 | 전체 트랜잭션 로그 대신 핵심 지표만 Firestore 기록 | Firestore 쓰기 70~80% 절감 |
| 비동기 처리 | 토큰 사용량 기록 등 후처리 fire-and-forget | 응답 레이턴시 없음 |
